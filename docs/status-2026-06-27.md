## ⏳ Open issues — remaining fixes needed

### 🔴 Critical (blocks real users)

#### F-4 / NEW-2 — 30% of host accounts are broken
**What:** `host_profiles` insert is missing a slug. `host_profiles.slug` is `NOT NULL` with no default, so any insert that omits it silently fails. Result: the user gets `role=host` in auth but no `host_profiles` row — so every dashboard page that calls `getHostProfile` redirects them out. They are a host who can't use any host tools.

**Confirmed broken in prod (2026-06-25):** 3 of 10 host accounts (30%) have `role=host` but no `host_profiles` row.

**Affected paths:**
- `app/(auth)/register/actions.ts` — email host signup
- `app/api/auth/upgrade-to-host/route.ts` — guest upgrading to host
- `app/(auth)/auth/callback/route.ts` — Google OAuth host path (unconfirmed)

**Fix needed:**
```ts
// Generate slug before inserting
const base = makeSlug(displayName) || "host";
let slug = base;
const { count } = await serviceClient.from("host_profiles")
  .select("id", { count: "exact", head: true }).eq("slug", slug);
if ((count ?? 0) > 0) slug = `${base}-${Math.random().toString(36).slice(2,6)}`;

const { data: existing } = await serviceClient.from("host_profiles")
  .select("id").eq("user_id", userId).maybeSingle();
if (!existing) await serviceClient.from("host_profiles")
  .insert({ user_id: userId, slug, display_name: name, email });
```
Also needed: one-off backfill for the 3 stranded prod hosts.

#### Issue 3 — Guest enquiry details exposed to other hosts
**What:** A host can see names, emails, and phone numbers of guests who enquired about *other* hosts' listings. RLS policy on `contact_requests` is too permissive.

**Fix needed:** Tighten the RLS policy so hosts can only read `contact_requests` where `listing_sanity_id` is in their own listings.

### 🟠 Incomplete features

#### D-1 — Hosts can't edit their own listings
Listing content lives in Sanity. Hosts have no way to edit it from the dashboard. Needs a host-scoped Sanity write API + `/dashboard/listings/[id]/edit` route.

#### D-2 — PMS property and marketplace listing are disconnected
A host who sets up a property in Property PMS sees "0 listings" in My Listings. The two models (Supabase `properties` and Sanity `listing`) have no link. Needs a "Publish to marketplace" bridge action.

#### D-7 — "Analyse with AI" on /list always fails
The AI-assist step on the listing form returns 503 even with `ANTHROPIC_API_KEY` set. The model ID in the route may be retired. Also needs `FIRECRAWL_API_KEY` for website scraping.

**Fix needed:**
1. Add error logging to `app/api/list/ai-analyse/route.ts` to surface the real cause
2. Update model ID to `claude-sonnet-4-6`
3. Add `FIRECRAWL_API_KEY` to env for website scraping

#### D-8 — `/api/admin/listing-requests/[id]/note` always 500s
`admin_notes` column missing from `listing_requests` table. Needs migration 073:
```sql
ALTER TABLE listing_requests ADD COLUMN admin_notes text;
```
