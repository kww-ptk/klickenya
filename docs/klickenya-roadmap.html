Strategic Principles

Build philosophy

The single biggest mistake marketplace founders make is building too much before launching. Klickenya follows three non-negotiable principles.

🌱
Ship lean, rank fast
V1 is a content + directory site. No booking engine yet — just a contact form. Goal is Google indexing, organic traffic, and host sign-ups before a single transaction.
V1 Priority
🏗
Structure first, features second
The database schema, URL architecture, and auth system are designed from day 1 to support the full marketplace — even if those features ship months later.
Day 1 Design
⚡
Content is the moat
Every listing, blog post, and destination page is a SEO asset. The content system is the product in V1. A listing added today should rank in 3 months.
SEO First
🔄
Upgrade paths, not rewrites
The contact form today becomes a booking widget tomorrow — same component, same URL, same database row. No migrations, no breaking changes, just new fields activated.
V2 Upgrade
🇰🇪
Kenya-native from day 1
M-Pesa as primary payment, Swahili copy in all transactional emails, KES as the currency unit, and local phone numbers as the primary contact field.
V2 Payment
🤖
AI-ready from day 1
The database schema, API design, and content model are built to plug in LLMs without migrations. pgvector is enabled on day 1. Webhooks fire on every event. GHL connects with a single env var.
Architectural Decision
📐
Multi-tenant ready
Every database table includes an owner_id from day 1. When public listing creation launches in V3, no schema changes are required.
V3 Scale
⚠️
The golden rule of V1
If a feature isn't needed to rank on Google or onboard the first 100 hosts — it doesn't ship in V1. Every hour spent on booking logic in V1 is an hour not spent on SEO content and host acquisition.

Technology Decisions

The tech stack

Every choice below is made with three constraints: fast to build in V1, scales to millions of users in V4, and has a strong ecosystem in 2026.

Frontend

Next.js 14

App Router, SSG for listings (perfect SEO), ISR for fresh content. The only framework that does everything Klickenya needs.

UI

Tailwind + shadcn/ui

Design system already built. shadcn gives accessible, unstyled components we can style with the Klickenya brand tokens.

Backend

Next.js API Routes

No separate server needed in V1–V2. Upgrade to standalone Node/Express only if API load demands it in V4.

Database

Supabase (Postgres)

Managed Postgres + auth + storage + realtime + row-level security — all in one. Free tier covers V1 entirely.

Auth

Supabase Auth

Email/password, Google OAuth, magic links — all built in. Handles admin-only in V1, public sign-ups in V3 with zero code changes.

CMS

Sanity.io

The page editor, blog system, and listing templates all live here. Non-developers can edit any page. GROQ queries feed the Next.js frontend.

Hosting

Vercel

Zero-config Next.js deployment, automatic preview URLs per branch, global CDN, and analytics built in.

Search

Postgres Full-Text Search

Built into Supabase — zero extra cost, zero extra service. tsvector columns + GIN indexes give fast full-text search across listings and blog posts. Upgrade to Typesense or Meilisearch only when needed at scale.

Payments

Stripe Connect

Split payments to hosts in V2. Kenya Stripe support confirmed. M-Pesa added via IntaSend or Pesapal bridge in V2.

M-Pesa

IntaSend SDK

The cleanest Daraja API wrapper for Node.js. Handles STK Push, C2B, B2C payouts. Much simpler than raw Daraja.

Email

Resend + React Email

Write emails as React components. Resend has a generous free tier and first-class Next.js integration.

Storage

Supabase Storage

Listing photos, blog images, host avatars. Built into Supabase — no separate S3 setup needed until V3+ scale.

Why Sanity for the CMS?

This is the most important decision for your "edit pages like a website builder" requirement. Sanity gives you:

✏️
Visual page editing
The Sanity Studio is a full visual editor. You can edit hero text, swap images, reorder sections — without touching code.
📝
Blog post management
Create, edit, schedule, and publish blog posts with rich text, images, tags, and SEO fields — entirely in the browser.
🏠
Listing management
Admin creates new listings directly in Sanity Studio. The listing appears on the site within seconds via ISR revalidation.
🔧
Structured content
Every field has a type. Title is always a string, price is always a number. No broken layouts from sloppy content entry.
💡
Sanity vs. a custom admin dashboard
Building a custom CMS (like WordPress) takes 6–8 weeks and requires ongoing maintenance. Sanity is free for up to 3 users, takes 2–3 days to configure, and gives you a professional editor immediately. For V1, it's the clear choice. In V3, Supabase-backed admin panels replace Sanity for dynamic data like bookings, users, and payments.

CREATE TABLE bookings (

CREATE TABLE availability (
  listing_id  uuid REFERENCES listings,

  status      availability_status,  -- available|blocked|booked

);

AI-Ready Architecture

Building AI-first from day 1

Every structural decision in Klickenya — the database schema, the API design, the content model — is made with AI integration in mind. Adding an AI feature in V2 should never require a migration or a rewrite. It should just be a new route that calls an LLM.

🤖
AI-ready ≠ AI now. It means AI never breaks anything later.
The V1 database, API, and content structure are designed so that plugging in Claude, GPT-4o, or any future LLM requires zero schema changes. The data is structured, typed, and described — exactly what LLMs need to operate reliably on your platform.

The 6 AI-readiness principles built into Klickenya from day 1

🏷
1. Everything is typed and described
Every database column, every API response, every Sanity field has a clear name and type. LLMs operate best on structured data with meaningful field names — not col_a, field_3.
Day 1
📋
2. Metadata-rich content model
Every listing has tags, category, city, county, amenities, description, and price. Every blog post has topic tags and a summary. This rich metadata makes vector embedding and semantic search trivial to add.
Day 1
🔌
3. Webhook-ready event system
Every significant action (new contact request, listing published, booking confirmed) fires a webhook. AI agents and automation tools (n8n, Make, GHL) subscribe to these webhooks to trigger workflows without polling.
Day 1
🗂
4. Atomic, single-responsibility API routes
Each API route does one thing. /api/listings returns listings. /api/contact handles enquiries. AI agents can call individual endpoints without needing to understand the whole system.
Day 1
🧠
5. LLM-friendly field naming
Field names like listing_type, price_per_night, check_in_date, max_guests are self-explanatory to any LLM. This means AI can query, summarise, and act on your data without a custom prompt for every field.
Day 1
🧩
6. Embedding-ready text columns
Listings and blog posts have a dedicated embedding_text virtual field (title + description + tags + city concatenated). When you add vector search, you embed this field — no data restructuring needed.
Day 1

Technical integration — how to connect GHL

// apps/web/lib/integrations/ghl.ts
// Simple, reliable GHL webhook sender — no SDK needed

const GHL_WEBHOOK_URL = process.env.GHL_INBOUND_WEBHOOK_URL

export async function sendToGHL(eventType: string, payload: Record<string, unknown>) {
  if (!GHL_WEBHOOK_URL) return  // silently skip if not configured (V1)

  try {
    await fetch(GHL_WEBHOOK_URL, {
      method: 'POST',
      headers: { 'Content-Type': 'application/json' },
      body: JSON.stringify({
        event: eventType,
        source: 'klickenya',
        timestamp: new Date().toISOString(),
        ...payload
      })
    })
  } catch (err) {
    console.error('GHL webhook failed:', err)
    // Never throw — GHL failure must not break the user flow
  }
}

// Usage in /api/contact/route.ts
await sendToGHL('contact.received', {
  name: data.fullName,

Bookings · V2

iCal sync — booking calendar integration

Hosts already list on Airbnb, Booking.com, and other platforms. iCal sync is the feature that makes Klickenya compatible with their existing workflow — and prevents the double bookings that destroy host trust.

📅
iCal sync is a trust feature, not a technical feature
A double booking — a guest arriving to find the listing occupied — is catastrophic for a marketplace. iCal sync prevents it by automatically blocking dates on Klickenya whenever a booking is made elsewhere. Without it, hosts who are on multiple platforms won't trust Klickenya with serious listings.

How iCal sync works

TWO-WAY SYNC ARCHITECTURE

IMPORT (other platforms → Klickenya)
─────────────────────────────────────
Host pastes Airbnb iCal URL into Klickenya listing settings
                    │
                    ▼
         Cron job runs every 2 hours
         GET [airbnb_ical_url]  →  parse .ics file
         Extract VEVENT blocks (DTSTART, DTEND, SUMMARY)
                    │
                    ▼
         Upsert into `availability` table:
         status = 'blocked', source = 'airbnb_ical'
         (existing Klickenya bookings are never overwritten)

EXPORT (Klickenya → other platforms)
─────────────────────────────────────
Klickenya generates a unique .ics feed per listing:
GET /api/listings/[id]/calendar.ics
         │
         ▼
Returns valid iCal file with:
- One VEVENT per confirmed Klickenya booking
- DTSTART / DTEND / SUMMARY ("Reserved via Klickenya")
- Updated automatically on every booking change

Host pastes this URL into Airbnb / Booking.com
→ Those platforms block the dates automatically

🔗

New table external_calendars: listing_id, platform (airbnb/booking/vrbo/manual), ical_url, last_synced_at, is_active. One listing can import from multiple platforms.
⚙️
Sync cron job
Vercel Cron at /api/cron/sync-calendars runs every 2 hours. Fetches all active external calendars, parses with the ical.js library, upserts blocked dates to the availability table.

GET /api/listings/[id]/calendar.ics — generates a valid iCal file from confirmed bookings. Uses ical-generator npm package. One unique URL per listing. Hosts share this with other platforms.

Before any booking is confirmed, a DB transaction checks the availability table. If any date in the range has status ≠ 'available', the booking is rejected. This check happens at DB level, not app level.

Auth architecture — role-based access control
Supabase Row Level Security (RLS) enforces permissions at the database level — not just in the UI. Even if someone bypasses the frontend, they cannot read data they don't own. admin role sees everything. host role sees only their own listings and bookings. guest role sees only their own bookings.
-- RLS policy example: hosts can only see their own listings
CREATE POLICY "Hosts see own listings"
  ON listings
  FOR ALL
  USING (
    owner_id = auth.uid()
    OR
    (SELECT role FROM users WHERE id = auth.uid()) = 'admin'
  );

Every AI tool — whether it's a Claude agent, a GHL workflow, an n8n automation, or a future custom model — will consume Klickenya's data through structured APIs and webhooks. The event system — how everything connects

Every significant action in Klickenya emits a structured event to a central webhook dispatcher. This is the nervous system of the platform — AI agents, GHL, n8n, and email automations all listen here.

// lib/events/dispatcher.ts — emit structured events from anywhere in the app
type KlickenyaEvent =
  | { event: 'listing.published';   payload: Listing }
  | { event: 'listing.updated';    payload: Listing }
  | { event: 'contact.received';   payload: ContactRequest & { listing: Listing } }
  | { event: 'booking.created';    payload: Booking & { listing: Listing; guest: User } }
  | { event: 'booking.confirmed';  payload: Booking }
  | { event: 'booking.cancelled';  payload: Booking & { reason: string } }
  | { event: 'review.posted';      payload: Review & { listing: Listing } }
  | { event: 'host.verified';      payload: User }
  | { event: 'payment.completed';  payload: { booking: Booking; method: 'card' | 'mpesa' } }

export async function emit(event: KlickenyaEvent) {
  // 1. Log to Supabase events table (audit trail, queryable by AI)
  await supabase.from('events').insert({ type: event.event, payload: event.payload })

  // 2. Fan out to all registered webhook subscribers
  const { data: subscribers } = await supabase
    .from('webhook_subscriptions')
    .select('*')
    .contains('events', [event.event])

  await Promise.allSettled(
    subscribers.map(sub => fetch(sub.url, {
      method: 'POST',
      headers: { 'Content-Type': 'application/json', 'X-Klickenya-Event': event.event },
      body: JSON.stringify(event.payload)
    }))
  )
}

LLM-Friendly Design

LLM layer & data design

LLMs don't read documentation — they read data. Every decision here is about making Klickenya's data maximally useful to any AI model that queries or modifies it.

What makes data LLM-friendly

✓ LLM-friendly patterns
  • Enums for all categorical fields (type, status, role)
  • ISO 8601 dates everywhere (2026-03-15T14:00:00Z)
  • Prices always in smallest unit (KES integer, never "4.8k")
  • Addresses structured: city + county + country separately
  • Tags as flat string arrays, not nested categories
  • Every record has created_at and updated_at
  • Rich text in Portable Text format (structured JSON, not HTML)
  • Photos as full URLs, not relative paths
  • Human-readable slugs as natural-language identifiers
✗ LLM-hostile patterns (never do these)
  • Free-text fields where enums should be used
  • Prices stored as formatted strings ("KSh 4,800/night")
  • Dates as Unix timestamps without ISO fallback
  • Nested JSON blobs with inconsistent shapes
  • HTML stored in database columns
  • Implicit relationships without foreign keys
  • Magic numbers without named constants
  • Computed fields that aren't denormalized

Semantic search with pgvector (V2)

-- supabase/migrations/004_vectors.sql
CREATE EXTENSION IF NOT EXISTS vector;

ALTER TABLE listings ADD COLUMN embedding vector(1536);
CREATE INDEX ON listings USING ivfflat (embedding vector_cosine_ops);

-- Match listings semantically
CREATE OR REPLACE FUNCTION match_listings(
  query_embedding vector(1536),
  match_threshold float DEFAULT 0.7,
  match_count int DEFAULT 10
) RETURNS SETOF listings AS $$
  SELECT * FROM listings
  WHERE 1 - (embedding <<=> query_embedding) > match_threshold
    AND status = 'published'
  ORDER BY embedding <<=> query_embedding
  LIMIT match_count;
$$ LANGUAGE sql;

Anti-double-booking guarantee

-- The only way to guarantee no double bookings is at the database level.
-- Application-level checks can race. This cannot.

CREATE OR REPLACE FUNCTION book_listing(
  p_listing_id uuid, p_check_in date, p_check_out date, p_guest_id uuid
) RETURNS uuid AS $$
DECLARE
  booking_id uuid;
BEGIN
  -- Lock the listing row for this transaction
  PERFORM id FROM listings WHERE id = p_listing_id FOR UPDATE;

  -- Check no dates are already blocked/booked
  IF EXISTS (
    SELECT 1 FROM availability
    WHERE listing_id = p_listing_id
      AND date BETWEEN p_check_in AND p_check_out - 1
      AND status != 'available'
  ) THEN
    RAISE EXCEPTION 'dates_unavailable';
  END IF;

  -- Insert booking and block all dates atomically
  INSERT INTO bookings (listing_id, guest_id, check_in, check_out, status)
  VALUES (p_listing_id, p_guest_id, p_check_in, p_check_out, 'confirmed')
  RETURNING id INTO booking_id;

  INSERT INTO availability (listing_id, date, status, source)
  SELECT p_listing_id, generate_series(p_check_in, p_check_out - 1, '1 day')::date, 'booked', 'klickenya'
  ON CONFLICT (listing_id, date) DO UPDATE SET status = 'booked';

  RETURN booking_id;
END;
$$ LANGUAGE plpgsql;
