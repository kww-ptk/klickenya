## 🔴 NEW-1 — Confirmation emails are not delivered (Supabase email rate-limited)

**Symptom:** On `klickenya.com/register`, signup **creates the account** and shows
"Check your email," but **no confirmation email arrives**, so the user can't
activate/log in.

**Investigation (read-only, against prod Supabase):** checked the auth records of
recent signups —
- Latest email/password signup (2026-06-23 16:35): `confirmation_sent_at` **is
  stamped**, `email_confirmed_at` is null → Supabase *recorded* a send, but the
  email never arrived.
- An earlier signup the same day (09:40): both stamped → that one's email **did**
  arrive and confirmed.
- Google signups: no confirmation needed (instant).

**Root cause:** Supabase Auth is using its **built-in/default email service**,
which is heavily rate-limited (only a couple of emails per hour) and explicitly
**not for production**. Once the limit is hit (easy with a handful of signups +
testing), Supabase stamps `confirmation_sent_at` but the message is **silently
dropped**. The "worked at 09:40, failed at 16:35" pattern is the classic
rate-limit signature.

**Important:** the app's **Resend** integration does NOT send these. Resend only
sends *application* emails (bookings, enquiries). **Supabase Auth emails go through
Supabase's own email/SMTP config**, which is currently the default service.

**Fix (Supabase dashboard config — not code):** point Supabase Auth at the existing
Resend account via custom SMTP.
- **Supabase → Authentication → Emails → SMTP Settings → Enable Custom SMTP:**
  - Host: `smtp.resend.com`
  - Port: `465`
  - Username: `resend`
  - Password: the Resend API key (`re_…`, same as `RESEND_API_KEY`)
  - Sender email: `noreply@klickenya.com` (must be on the Resend-verified
    `klickenya.com` domain — already verified, since `bookings@klickenya.com` works)
  - Sender name: `Klickenya`
- Then **Authentication → Rate Limits**: raise the email limit if still low.

**Relationship to yesterday's F-1 fix:** F-1 fixed the *link inside* the email
(domain). NEW-1 is about the email *being sent at all*. Both are needed for
registration to fully work, plus F-1 must actually be **deployed to production**
(verify the `dev → main` PR was merged).

## 🔴 NEW-2 — 30% of production hosts have no `host_profile` (live impact of F-4 / R-4)

**Measured 2026-06-25, read-only against prod:**
- **118 guests · 10 hosts · 2 admins**
- **3 of 10 host users (30%) have `role=host` but NO `host_profiles` row:**
  - `info***` (d29132b2)
  - `info***` (b54171e9)
  - `jule***` (b45cd441)

**Impact (live):** a host with no `host_profile` is **redirected out of every
dashboard page that calls `getHostProfile`** (My Listings, Enquiries, Stats,
Property PMS gates, etc.) — they cannot use the host tools at all. These are real,
stranded accounts in production, not a hypothetical.

**This upgrades a prior "needs-verify" flag to CONFIRMED.** The fix plan (F-4)
noted *"possibly affects the Google path too."* The data confirms the broken
accounts come from **more than one provisioning path**, so this is **not only** the
email-signup bug — multiple host-creation paths (R-4) fail to create the profile.

**Root cause (same underlying schema issue as F-4):**
- `host_profiles.user_id` has **no unique constraint** → `upsert(onConflict:
  "user_id")` fails `42P10`.
- `host_profiles.slug` is **NOT NULL, no default** → any insert omitting a slug
  fails `23502`.
Any host-creation path that doesn't generate a slug and insert correctly leaves a
profile-less host.

**Fix direction:**
1. **Consolidate** all host provisioning into one `provisionHost(userId, name,
   email)` helper (generate a unique slug + insert the profile) used by the OAuth
   callback, email register, claim approve, and list approve (kills R-4 drift).
2. **One-off data repair:** backfill `host_profiles` for the 3 stranded hosts
   (generate a slug from their name) so they can access the dashboard.
